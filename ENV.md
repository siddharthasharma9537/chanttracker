## Configuration by Environment

### 1. Local Development

**Files:** `.env.local` (root directory)

```bash

NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=sb_publishable_O6pzpR_EmGHBumqrF6nX9A_1Ge2hPpS
VERCEL_OIDC_TOKEN=<auto-generated by Vercel CLI>
```

**How it works:**
- Next.js reads `.env.local` automatically during `pnpm dev`
- Variables are available in both server and client code
- `VERCEL_OIDC_TOKEN` is auto-generated when you run `vercel link` or `vercel login`

## How Code Accesses Variables

### Next.js Client & Server

All files can access these variables since they're prefixed with `NEXT_PUBLIC_`:

```typescript
// packages/api/index.js
const URL = process.env.NEXT_PUBLIC_SUPABASE_URL
const KEY = process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY

export const supabase = createClient(URL, KEY)
```

```typescript
// apps/web/src/lib/supabase/client.ts
export function createClient() {
  return createBrowserClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!
  )
}
```

## Adding New Environment Variables

### Step 1: Add to `.env.example`

```bash
# Document the new variable with its purpose
NEXT_PUBLIC_NEW_VAR=example_value_here
```

### Step 2: Add to `.env.local` (local development)

```bash
NEXT_PUBLIC_NEW_VAR=actual_value
```

### Step 3: Add to Vercel

```bash
# Production
vercel env add NEXT_PUBLIC_NEW_VAR production --value "actual_value" --yes

# Development
vercel env add NEXT_PUBLIC_NEW_VAR development --value "actual_value" --yes

# Preview (all feature branches)
vercel env add NEXT_PUBLIC_NEW_VAR preview <branch-name> --value "actual_value" --yes
```

### Step 4: Use in code

```typescript
const newVar = process.env.NEXT_PUBLIC_NEW_VAR
```

## Security Notes

⚠️ **Never commit `.env.local` to GitHub** — it contains actual values

✅ **Safe to commit:** `.env.example` — contains only placeholder values

✅ **These variables are intentionally public:**
- Publishable keys (starting with `sb_publishable_`) are designed to be exposed
- They work alongside Supabase RLS (Row-Level Security) policies
- Service role keys (never included) provide backend-only access

🔐 **Vercel automatically encrypts values** in the dashboard

## Troubleshooting

### Variables not loading in local development

1. **Check `.env.local` exists:**
   ```bash
   ls -la .env.local
   ```

2. **Verify naming** — must start with `NEXT_PUBLIC_` for client access

3. **Restart dev server:**
   ```bash
   pnpm dev
   ```

### Variables not available in Vercel deployment

1. **Verify they're added to correct environment:**
   ```bash
   vercel env list
   ```

2. **Check branch name** — Preview vars are branch-specific

3. **Redeploy after adding** — Vercel rebuilds with new vars

### "Missing Supabase environment variables" error

**Cause:** `NEXT_PUBLIC_SUPABASE_URL` or `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` not set

**Fix:**
1. Add to `.env.local` (local dev) or Vercel (production)
2. Ensure correct `NEXT_PUBLIC_` prefix
3. Restart dev server or redeploy
